2 ЗАВДАННЯ
Дослідження структурного патерну проєктування програмного забезпечення Facade: аналіз його поняття, структури та компонентів, практичної реалізації, переваг і недоліків, а також порівняння з суміжними патернами проєктування.
3 ОПИС ВИКОНАНОЇ РОБОТИ
У сучасній розробці програмного забезпечення застосування перевірених архітектурних рішень є невід'ємною частиною побудови якісних систем. Патерни проєктування - це типові рішення поширених проблем, що виникають у процесі проєктування програмного забезпечення. Слід зазначити, що патерн не є готовим фрагментом коду, придатним до прямого використання, - це загальний шаблон або концепція вирішення певної задачі, яку необхідно адаптувати під конкретні умови. Патерни формалізують накопичений досвід розробників і дозволяють передавати кращі практики у вигляді чітко описаних підходів. Серед основних цілей їх застосування - повторне використання перевірених рішень, покращення читабельності та підтримуваності коду, спрощення комунікації між членами команди, уникнення типових архітектурних помилок, а також прискорення розробки.
Відповідно до загальноприйнятої класифікації, усі патерни проєктування поділяються на три групи. Породжувальні патерни відповідають за гнучке та безпечне створення об'єктів і визначають механізм їх інстанціювання (Singleton, Factory Method, Builder). Структурні патерни описують способи організації об'єктів у більші структури (Facade, Adapter, Decorator, Proxy). Поведінкові патерни визначають ефективні способи взаємодії між об'єктами та розподіл відповідальності між ними (Observer, Strategy, Command).
Предметом розгляду є структурний патерн Facade. Назва походить від архітектурного поняття «фасад будівлі»: ззовні ми бачимо просту й охайну поверхню, тоді як за нею прихована складна інженерна конструкція. Аналогічний принцип застосовується і в програмуванні: Facade надає клієнту спрощений інтерфейс до складної підсистеми або набору класів, приховуючи внутрішню реалізацію. Патерн приховує складність підсистеми, забезпечує єдину точку входу до неї, зменшує залежності між клієнтським кодом та компонентами підсистеми, а також суттєво спрощує роботу зі складними API та зовнішніми фреймворками.
Структура патерну містить чотири ключові компоненти. Facade - основний клас зі спрощеним інтерфейсом, який знає, які підсистеми відповідають за виконання тих чи інших запитів, і делегує їм відповідні виклики. Subsystem classes - класи підсистеми, що містять складну бізнес-логіку; вони не знають про існування фасаду та можуть використовуватись як безпосередньо, так і через нього. Client - клієнтський код, який взаємодіє виключно з фасадом, не маючи прямого доступу до підсистеми. Additional Facade - додатковий фасад, який може створюватись для виокремлення непов'язаних між собою функцій та уникнення перевантаження основного класу.
Визначено випадки, коли доцільно застосовувати патерн Facade: коли підсистема містить значну кількість класів і є складною для розуміння; коли підсистема постійно ускладнюється в процесі розробки й підтримки; коли необхідно структурувати систему на окремі шари абстракції; коли потрібно спростити залежності між різними підсистемами; коли клієнтський код надмірно залежить від деталей реалізації підсистеми.
Для ілюстрації практичного застосування розглянуто приклад системи домашнього кінотеатру. Без застосування Facade клієнт змушений вручну керувати кожним пристроєм окремо: вмикати підсилювач, встановлювати рівень гучності, налаштовувати DVD-програвач, вмикати проєктор та регулювати освітлення. При використанні патерну достатньо одного виклику методу watchMovie - клас HomeTheaterFacade самостійно виконає всі необхідні дії, делегуючи їх відповідним підсистемам.
Серед переваг патерну Facade слід відзначити: надійну ізоляцію клієнтського коду від складності підсистеми, суттєве зменшення кількості залежностей між компонентами, спрощення міграції або заміни реалізації підсистеми без змін у клієнтському коді, а також покращення загальної читабельності архітектури. Разом із тим існують певні недоліки: головний ризик полягає у перетворенні фасаду на «Бог-об'єкт» - клас, який знає забагато і виконує забагато функцій; крім того, може виникнути надмірна залежність усього коду від одного класу, а налагодження системи може ускладнитись.
Проведено порівняння Facade з суміжними структурними патернами. Adapter адаптує існуючий інтерфейс під очікуваний, здійснюючи «переклад» між двома несумісними інтерфейсами, тоді як Facade створює принципово новий спрощений інтерфейс для цілої підсистеми. Mediator організує двосторонню комунікацію між компонентами, тоді як Facade лише спрощує односторонній доступ. Proxy зберігає той самий інтерфейс, що й реальний об'єкт, тоді як Facade надає новий спрощений. Decorator розширює існуючий інтерфейс новою функціональністю, тоді як Facade замінює складний інтерфейс на простіший.

4 ВИСНОВКИ
У ході роботи було досліджено структурний патерн проєктування Facade: розглянуто його поняття та місце у загальній класифікації патернів, детально проаналізовано структуру та чотири ключові компоненти, визначено типові сценарії застосування, розглянуто практичну реалізацію на прикладі системи домашнього кінотеатру, а також проведено порівняння з суміжними патернами Adapter, Mediator, Proxy та Decorator.
Facade є одним із найпростіших та найпоширеніших структурних патернів проєктування. Він ефективно вирішує проблему складності великих підсистем та зовнішніх бібліотек і широко застосовується у реальних проєктах - у фреймворках, SDK, бібліотеках та клієнтах зовнішніх API. Його використання дозволяє дотримуватися принципів єдиної відповідальності та слабкого зв'язування компонентів, що значно спрощує довгостроковий супровід і масштабування проєктів. Рекомендується застосовувати цей патерн у ситуаціях, коли складність підсистеми суттєво перевищує реальні потреби клієнтського коду.

Програмний код
В.1 Структура патерну Facade
1  // Підсистеми
2  class Amplifier {
3    on() {
4      console.log('Amplifier on');
5    }
6    setVolume(v) {
7      console.log('Volume: ${v}');
8    }
9  }
10 class DVDPlayer {
11   play(movie) {
12     console.log(`Playing: ${movie}`);
13   }
14 }
15 class Projector {
16   on() {
17     console.log('Projector on');
18   }
19 }
20 // Facade
21 class HomeTheaterFacade {
22   constructor(amp, dvd, proj) {
23     this.amp = amp;
24     this.dvd = dvd;
25     this.proj = proj;
26   }
27 }

В.2 Приклад реалізації
1  // Без Facade - ручне управління
2  const amp = new Amplifier();
3  const dvd = new DVDPlayer();
4  const proj = new Projector();
5  amp.on();
6  amp.setVolume(10);
7  proj.on();
8  dvd.play('Inception');
9  // З Facade - один виклик
10 class HomeTheaterFacade {
11   watchMovie(movie) {
12     this.amp.on();
13     this.amp.setVolume(10);
14     this.proj.on();
15     this.dvd.play(movie);
16   }
17 }
18 const theater = new HomeTheaterFacade(amp, dvd, proj);
19 theater.watchMovie('Inception');

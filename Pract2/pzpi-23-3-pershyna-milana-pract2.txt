2 ЗАВДАННЯ
Дослідження архітектури програмної системи Figma: аналіз її тришарової структури, клієнтської та серверної частин, організації зберігання даних, механізму синхронізації в реальному часі, а також порівняння з конкурентними рішеннями.
3 ОПИС ВИКОНАНОЇ РОБОТИ
Figma - це хмарна платформа для проєктування інтерфейсів та прототипування, що дозволяє командам співпрацювати в режимі реального часу безпосередньо у браузері. Платформу засновано у 2012 році Діланом Філдом та Евансом Воллесом. Figma є однією з перших повноцінних дизайн-систем, що не вимагає встановлення жодного програмного забезпечення. Серед ключових особливостей - повна браузерна доступність, хмарне зберігання та синхронізація файлів, можливість одночасної роботи кількох користувачів, кросплатформеність на Windows, macOS та Linux, а також відкритий API для інтеграцій та плагінів.
Архітектура Figma складається з трьох основних шарів. Клієнтський шар (Client Layer) включає браузерний або desktop-застосунок, побудований на WebAssembly та Canvas і WebGL. Серверний шар (Server Layer) реалізований за допомогою мікросервісів на Node.js та Go з REST API і WebSocket-з'єднаннями. Шар даних (Data Layer) охоплює PostgreSQL для реляційних даних, Redis для кешування і черг, а також Amazon S3 для файлових активів. Такий поділ на три рівні забезпечує чітке розмежування відповідальності та можливість незалежного масштабування кожного рівня.
Клієнтська частина Figma побудована на нетиповому принципі: основний рушій рендерингу написаний мовою C++ і скомпільований у WebAssembly, що дозволяє досягти продуктивності, близької до нативних застосунків, прямо у браузері. Для відображення графіки використовується Canvas 2D та WebGL з підтримкою GPU-прискорення. UI-шар з панелями інструментів та діалогами реалізований на React.js, а вся бізнес-логіка написана на TypeScript. Така комбінація технологій забезпечує одночасно високу продуктивність та зручність розробки.
Серверна частина побудована за мікросервісною архітектурою. Основні мови розробки - Node.js та Go, причому Go використовується для критичних за продуктивністю сервісів. REST API обслуговує стандартні операції над ресурсами - файлами, проєктами та користувачами. WebSocket забезпечує передачу змін у реальному часі між учасниками сесії. Для злиття паралельних змін від кількох користувачів реалізовано алгоритм Operational Transformation. Внутрішня комунікація між мікросервісами відбувається через gRPC, а оркестрація контейнерів реалізована на базі AWS ECS та Kubernetes.
Шар зберігання даних складається з чотирьох ключових компонентів. PostgreSQL - основна реляційна СУБД, що зберігає метадані файлів, дані користувачів, налаштування дозволів та версійність. Redis — in-memory сховище для кешування, черг повідомлень та активних сесій. Amazon S3 - об'єктне хмарне сховище для зображень, експортованих файлів та бінарних активів. Для зберігання станів документів використовуються структури CRDTs (Conflict-free Replicated Data Types), що дозволяють зливати паралельні зміни без конфліктів.
Одна з ключових особливостей Figma - синхронізація в реальному часі, що відбувається у чотири кроки. По-перше, користувач вносить зміну, яка одразу застосовується локально - так звана оптимістична модель. По-друге, зміна передається на сервер синхронізації через WebSocket. По-третє, сервер за допомогою алгоритму Operational Transformation трансформує операцію відносно інших паралельних змін і розсилає її всім учасникам. По-четверте, кожен клієнт отримує трансформовану операцію та оновлює свій стан документа. Такий підхід гарантує узгодженість даних у всіх учасників сесії.
Серед переваг архітектурних рішень Figma варто виділити: хмарну модель без необхідності встановлення та оновлення програмного забезпечення, безконфліктну спільну роботу в реальному часі, кросплатформеність через браузер та WebAssembly, високу продуктивність рендерингу завдяки C++ і WASM, а також масштабованість мікросервісної архітектури на AWS. Серед недоліків слід зазначити: залежність від інтернет-з'єднання та обмежений офлайн-режим, закриту архітектуру без можливості самостійного розгортання, питання конфіденційності даних при хмарному зберіганні, а також ризик vendor lock-in.
Проведено порівняння архітектури Figma з конкурентами. Sketch є нативним macOS-застосунком із локальним зберіганням, тоді як Figma перевершує його завдяки хмарній співпраці та кросплатформеності. Adobe XD зупинив активний розвиток у 2023 році, тоді як Figma має ширшу спільноту та кращу підтримку. Penpot — відкрите рішення для самостійного розгортання, що виграє у питаннях контролю даних, але поступається Figma за функціональністю. Framer орієнтований на інтерактивні прототипи з кодом, тоді як Figma ефективніша для комплексних дизайн-систем і командної роботи.

4 ВИСНОВКИ
У ході роботи було досліджено архітектуру програмної системи Figma: розглянуто загальну характеристику платформи, проаналізовано тришарову архітектуру та технологічний стек клієнтської і серверної частин, досліджено організацію шару зберігання даних, детально розглянуто механізм синхронізації в реальному часі на основі алгоритму Operational Transformation, а також проведено порівняння з конкурентними рішеннями.
Figma є прикладом інноваційної хмарної архітектури, що поєднала WebAssembly, мікросервіси та OT-алгоритми в єдиній системі. Використання C++ і WASM дозволило досягти нативної продуктивності у браузері, мікросервісна архітектура забезпечує горизонтальне масштабування та відмовостійкість, а комбінація AWS S3, PostgreSQL та Redis гарантує надійність і швидкість доступу до даних. Архітектура Figma стала еталоном для нового покоління колаборативних веб-застосунків.
